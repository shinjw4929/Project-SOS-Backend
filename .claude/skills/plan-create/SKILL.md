## 역할

사용자의 요구사항을 받아 `docs/Plans/계획-작성-가이드.md`의 형식을 따르는 구현 계획을 생성한다.

$ARGUMENTS가 있으면 해당 내용을 변경 대상/문제/목표로 반영한다.

## 계획 생성 절차

### 0단계: 가이드 문서 로드

`docs/Plans/계획-작성-가이드.md`를 읽어 플랜 산출물 구조와 작성 규칙을 확인한다.

### 1단계: 요구사항 정리

사용자로부터 다음 4가지를 확보한다. $ARGUMENTS에서 추출할 수 없는 항목은 사용자에게 질문한다.

| 항목 | 설명 |
|---|---|
| **변경 대상** | 어떤 모듈/서비스/설정이 변경되는지 |
| **현재 문제** | 지금 왜 안 되는지, 왜 바꿔야 하는지 |
| **원하는 결과** | 변경 후 어떻게 동작해야 하는지 |
| **제약 조건** | 하위 호환, 성능 제약, 인프라 제약 등 |

### 2단계: AS-IS 조사

**implement 탐색 결과가 인계된 경우**: $ARGUMENTS에 `## implement 탐색 결과 (인계)` 섹션이 포함되어 있으면, 해당 내용을 AS-IS의 기초 자료로 활용한다. 이미 파악된 docs/레퍼런스/영향 범위는 재탐색하지 않고, 인계 내용에서 부족한 부분만 추가 조사한다.

**인계 없이 단독 실행하는 경우**: Explore agent를 활용하여 코드베이스를 처음부터 조사한다:

1. **Docs 참조**: `docs/Systems/` 폴더에서 변경 대상과 관련된 문서를 읽어 현재 시스템 구조 파악
2. **코드 탐색**: 변경 대상 모듈/클래스의 실제 코드를 읽어 현재 동작 방식 확인
3. **의존성 파악**: 변경 대상을 참조하는 다른 모듈/클래스를 Grep으로 탐색
4. **영향 범위**: 모듈 분포(common/room/chat/infra), Protobuf 메시지, Redis 키, Docker 설정 영향 확인

조사 결과를 바탕으로 AS-IS를 **코드 근거 기반**으로 작성한다. 추측 금지.

### 3단계: TO-BE 설계

AS-IS와 요구사항을 기반으로 TO-BE를 설계한다:

1. 변경/추가/삭제 항목을 구체적으로 명시
2. 모듈 배치 결정 (common/room/chat/infra)
3. Protobuf 메시지 설계 (필요 시)
4. Redis 키 구조 설계 (필요 시)
5. AS-IS vs TO-BE 비교표 작성

### 4단계: Phase 분할

작업을 롤백 가능한 단위로 Phase를 나눈다:

- **Phase 0**: 사전 조사가 필요한 경우에만 포함 (2단계에서 충분히 조사했다면 생략 가능)
- **Phase N**: 각 Phase는 독립적으로 롤백 가능해야 함
- 각 Phase에 테스트 요구사항과 완료 기준을 포함
- 병렬 실행 가능한 Task는 subagent 구성을 명시

### 5단계: 계획 문서 생성

`docs/Plans/[기능명]/` 디렉토리를 생성하고 다음 파일을 작성한다:

1. `orchestration.md` - 오케스트레이션 (중심 문서, Phase 체크리스트 포함)
2. `phase-N-[제목].md` - 각 Phase 상세 파일
3. `execution-log.md` - 빈 실행 기록 템플릿 (헤더만)

기능명은 사용자 요구사항에서 추출하되, 간결하고 명확한 한국어 또는 영어로 명명한다.

### 6단계: 자동 리뷰

계획 생성 후 `/review-plan [기능명]` 스킬을 호출하여 자동 검증한다.

review-plan가 "재계획 필요"로 판정하면 내부적으로 `/plan-edit`를 호출하여 수정한 뒤 재검토까지 수행한다 (review-plan 내부 루프 최대 3회).

review-plan가 "승인 가능"을 반환하면 다음 단계로 진행한다.

### 7단계: 계획 요약본 저장

검증 완료된 계획의 요약을 `docs/Plans/[기능명]/summary.md`에 저장한다:
- 문제 정의 핵심
- Phase 구성 요약
- 예상 영향 범위
- 자동 리뷰 통과 여부 (N회차에 승인)

### 8단계: 사용자에게 결과 보고

생성된 문서 목록과 각 문서의 핵심 내용을 간략히 보고한다.
다음 단계로 `/plan-execute [기능명]`을 안내한다.

## 주의사항

- AS-IS는 반드시 코드를 직접 읽고 작성한다. 문서만 참조하지 않는다.
- Phase 파일의 작업 목록은 구현 가능한 수준으로 구체적으로 작성한다 (파일명, 클래스명, 함수명 포함).
- 기존 `docs/Plans/` 아래에 동일 기능의 계획이 있으면 사용자에게 알리고 덮어쓸지 확인한다.
- CLAUDE.md의 Development Guidelines를 따른다 (RAII, smart pointer, strand 등).
