## 역할

코드를 수정하거나 새로 작성할 때, 코드베이스에서 유사한 기존 패턴을 먼저 탐색한 뒤 동일한 구조/네이밍/관용구를 따르는 코드를 생성한다.

$ARGUMENTS가 있으면 해당 내용을 구현 대상/목표로 반영한다. 없으면 사용자에게 무엇을 구현할지 질문한다.

## 구현 절차

### 1단계: 요구사항 파악

$ARGUMENTS에서 다음을 추출한다:

| 항목 | 설명 |
|---|---|
| **구현 대상** | 무엇을 만들거나 수정하는지 (핸들러, 유틸리티, 서비스, 설정 등) |
| **동작 목표** | 완성된 코드가 어떻게 동작해야 하는지 |
| **수정 범위** | 새 파일 생성 vs 기존 파일 수정 |

추출할 수 없는 항목이 있어도 최대한 추론하여 진행한다. 핵심 정보가 부족한 경우에만 사용자에게 질문한다.

### 2단계: 기존 패턴 탐색 및 분석

`docs/Checklists/pattern-search-guide.md`를 따라 레퍼런스를 탐색하고 패턴을 추출한다.

1. **Docs 참조**: `docs/Systems/` 폴더에서 구현 대상과 관련된 문서를 읽어 현재 시스템 구조를 파악한다.
2. **유형별 탐색**: 가이드의 유형별 탐색 전략에 따라 레퍼런스 1~3개를 선정한다.
3. **패턴 추출**: 레퍼런스에서 구조, 네이밍, 에러 처리, 비동기 패턴을 추출한다.
4. **영향 범위 파악**: 탐색 과정에서 변경/생성이 필요한 파일 목록을 정리한다.

### 2.5단계: 규모 판단

2단계에서 파악한 변경/생성 대상 파일이 **3개 이상**이면, 계획 수립 전환을 사용자에게 제안한다.

**제안 형식:**

```
변경 규모가 큽니다 (대상 파일 N개).
| # | 파일 | 변경 유형 | 사유 |
|---|------|-----------|------|
| 1 | ...  | 신규/수정 | ...  |

계획을 먼저 세울까요? (→ /plan-create)
아니면 이대로 구현을 진행할까요?
```

- **사용자가 plan-create 전환을 선택한 경우**: 2단계에서 수집한 탐색 결과를 정리하여 `/plan-create`에 전달한다. 전달 형식:

  ```
  /plan-create [구현 목표 요약]

  ## implement 탐색 결과 (인계)
  - Docs 참조: [읽은 문서 목록과 핵심 내용]
  - 레퍼런스 파일: [선정한 레퍼런스와 추출한 패턴]
  - 영향 범위: [변경/생성 대상 파일 목록과 사유]
  ```

- **사용자가 그대로 구현을 선택한 경우**: 3단계로 진행한다.
- **대상 파일이 2개 이하**: 이 단계를 건너뛰고 3단계로 바로 진행한다.

### 3단계: 구현

추출한 패턴을 기반으로 코드를 작성한다. pattern-search-guide.md의 적용 원칙을 따른다.

**구현 규칙:**

1. **CLAUDE.md Development Guidelines 준수**: RAII, smart pointer, 스레드 안전성, 네이밍, 환경 변수 규칙을 따른다.
2. **최소 변경**: 요구된 기능만 구현한다. 주변 코드 리팩토링이나 추가 기능을 임의로 넣지 않는다.
3. **기존 코드 수정 시**: 해당 파일의 기존 스타일을 그대로 따른다. 수정 범위 밖의 코드를 건드리지 않는다.
4. **CMakeLists.txt 반영**: 새 소스 파일 추가 시 해당 모듈의 CMakeLists.txt에 반영한다.
5. **Protobuf 변경 시**: proto/ 파일 수정 + 빌드 시 코드젠 자동 실행 확인.

**병렬 구현 (독립 파일 2개 이상):**
- 서로 의존하지 않는 파일은 Agent를 활용하여 병렬로 작성한다.
- 각 Agent에게 레퍼런스 코드 + 구현 요구사항 + CLAUDE.md 규칙 요약을 전달한다.
- 같은 파일을 수정하는 작업은 반드시 순차 실행한다.

### 4단계: 검증

1. **컴파일 확인**: 소스 파일이 변경된 경우 `/build`로 빌드 검증을 권장한다.
2. **테스트**: 유틸리티/순수 함수를 구현한 경우 대응 테스트 작성 여부를 확인한다. 테스트가 필요하면 기존 테스트의 패턴을 레퍼런스로 삼아 작성한다.
3. **패턴 일관성 자가 점검**: 구현한 코드가 레퍼런스와 구조적으로 일관된지 확인한다.

### 5단계: 후처리 (자동)

구현 완료 후 다음 스킬을 자동 호출한다:

1. `/review-comments` - 변경된 파일의 주석 정합성 점검 및 수정
2. `/update-docs` - 변경사항에 대응하는 docs 문서 업데이트

### 6단계: 결과 보고

```
## 구현 완료

### 레퍼런스 패턴
| # | 파일 | 참조 이유 |
|---|------|-----------|
| 1 | ...  | ...       |

### 변경 사항
| 파일 | 변경 유형 | 요약 |
|------|-----------|------|
| ...  | 신규/수정 | ...  |

### 후속 권장
- (필요 시: /build, /test, /review-code)
```

## 주의사항

- **레퍼런스 필수**: 코드를 작성하기 전에 반드시 최소 1개의 레퍼런스를 탐색한다. 레퍼런스 없이 코드를 작성하지 않는다.
- **과잉 탐색 금지**: 레퍼런스는 최대 3개로 제한한다. 탐색에 지나치게 많은 시간을 쓰지 않는다.
- **기존 패턴 우선**: 프로젝트의 기존 패턴이 일반적인 모범 사례와 다르더라도, 일관성을 위해 기존 패턴을 따른다. 단, 패턴이 명백히 버그를 유발하는 경우는 사용자에게 알린다.
- **사실 기반**: 탐색 결과 유사한 패턴이 없으면, 없다고 보고하고 CLAUDE.md 규칙과 일반 C++ 관례에 따라 구현한다.
