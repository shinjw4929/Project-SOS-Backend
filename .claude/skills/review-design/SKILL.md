## 역할

코드 변경 또는 구현 계획이 `docs/Systems/` 문서와 `CLAUDE.md`에 정의된 시스템 설계 방향성과 일치하는지 검증한다.
코드 수정은 하지 않는다. 검증 결과만 출력한다.

$ARGUMENTS가 있으면 해당 내용을 검증 대상/관점으로 반영한다.

## 검증 절차

### 0단계: 설계 방향성 로드

다음을 병렬로 읽어 현재 시스템 설계 방향성을 파악한다:
- `CLAUDE.md`의 Architecture, Redis 키 구조, Port Assignment, Development Guidelines 섹션
- 검증 대상이 속한 모듈의 `docs/Systems/` 문서

문서가 없거나 비어있으면 사용자에게 알리고 중단한다.

### 1단계: 검증 대상 파악

검증 대상을 다음 순서로 결정한다:

1. **$ARGUMENTS가 플랜 경로/기능명**: 해당 계획의 orchestration.md + phase 파일을 읽는다.
2. **$ARGUMENTS가 구체적 설명**: 해당 내용을 검증 대상으로 삼는다.
3. **$ARGUMENTS 없음**: `git diff HEAD`로 현재 코드 변경사항을 검증 대상으로 삼는다.

### 2단계: 설계 방향성 대조

검증 대상을 시스템 설계 문서의 각 영역과 대조한다:

| 검증 항목 | 설계 기준 | 위반 예시 |
|-----------|-----------|-----------|
| **모듈 경계** | common/room/chat 분리, 공유 코드는 common | room에서 chat 직접 참조, 모듈 간 순환 의존 |
| **Redis 키 설계** | CLAUDE.md 키 패턴, TTL, 타입 규칙 | 미등록 키 패턴 사용, TTL 미설정, 타입 불일치 |
| **포트 할당** | CLAUDE.md Port Assignment 테이블 | 미등록 포트 사용, 기존 포트와 충돌 |
| **Protobuf 프로토콜** | proto/ 정의, Envelope 패턴 | Envelope 우회한 직접 직렬화, proto 미등록 메시지 |
| **내부 채널 설계** | Room<->Game, Room->Chat 채널 구조 | 채널 역방향 사용, 미정의 내부 프로토콜 |
| **인증/보안** | 토큰 검증 필수, Rate Limiting 필수 | 미인증 요청 허용, Rate Limit 미적용 |
| **스레드 안전성** | strand 기반 동시성, RAII 필수 | strand 미사용 공유 자원 접근, raw pointer |
| **로그 형식** | Vector 파싱 정규식 일치 형식 | 비표준 로그 형식, 카테고리 미지정 |
| **인프라 정합성** | Docker Compose 서비스/포트/의존성 | docker-compose.yml 미반영 설정 변경 |
| **설정 관리** | Config 클래스 접근, .env 관리 | 설정값 하드코딩, 환경변수 미사용 |

### 3단계: 검증 결과 출력

```
## 시스템 설계 검증 결과

### 검증 대상
- (코드 변경 / 구현 계획 / 구체적 설명)

### 일치 항목
- (설계 방향성과 부합하는 부분)

### 불일치 항목
| # | 설계 원칙 | 현재 구현/계획 | 불일치 내용 | 심각도 |
|---|-----------|---------------|-------------|--------|
| 1 | ...       | ...           | ...         | 위반/주의 |

### 미확인 항목
- (설계 문서에 아직 정의되지 않아 판단 불가한 부분)

### 최종 판단
(설계 부합 / 설계 불일치 / 설계 미정의)
```

**심각도 기준**:
- **위반**: 설계 원칙에 명시적으로 반하는 구현 (예: 모듈 순환 의존, 미인증 접근 허용)
- **주의**: 설계 의도와 미묘하게 어긋날 수 있는 구현 (예: 공유 코드가 common 대신 room에 위치)

### 4단계: 후속 안내

- **설계 부합**: 결과만 출력하고 종료.
- **설계 불일치**: 위반 항목을 나열하고, 설계에 맞는 대안을 제안한다. 코드 수정이 필요하면 직접 수정하지 않고 사용자에게 안내한다.
- **설계 미정의**: 해당 영역의 설계를 docs/Systems/ 문서에 추가할 것을 권장한다.

## 주의사항

- **설계 문서가 최우선**: 코드가 설계와 다르면, 설계가 맞는지 코드가 맞는지 판단하지 않는다. 불일치 사실만 보고하고 사용자가 판단한다.
- **코드 수정 금지**: 이 스킬은 검증만 수행한다. 수정이 필요하면 사용자에게 안내한다.
- **과잉 해석 금지**: docs/Systems/ 및 CLAUDE.md에 명시적으로 기술된 내용만 기준으로 삼는다. 문서에 없는 설계 의도를 추측하지 않는다.
- **미정의 영역 존중**: 문서에 정의되지 않은 영역은 위반으로 판단하지 않고 "미정의"로 분류한다.
