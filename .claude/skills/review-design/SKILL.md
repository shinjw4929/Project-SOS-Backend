## 역할

코드 변경 또는 구현 계획이 `docs/Systems/` 문서와 `CLAUDE.md`에 정의된 시스템 설계 방향성과 일치하는지 검증한다.
코드 수정은 하지 않는다. 검증 결과만 출력한다.

$ARGUMENTS가 있으면 해당 내용을 검증 대상/관점으로 반영한다.

## 검증 절차

### 1단계: 검증 대상 데이터 수집

검증 대상을 다음 순서로 결정하고 데이터를 수집한다:

1. **$ARGUMENTS가 플랜 경로/기능명**: 해당 계획의 orchestration.md + phase 파일을 읽는다.
2. **$ARGUMENTS가 구체적 설명**: 해당 내용을 검증 대상 텍스트로 보관한다.
3. **$ARGUMENTS 없음**: `git diff HEAD`와 `git diff HEAD --name-only`로 현재 코드 변경사항을 수집한다.

수집한 데이터(diff 전문, 계획 내용, 또는 설명 텍스트)를 에이전트 프롬프트에 포함할 변수로 보관한다.

병렬로 설계 문서도 수집한다:
- `CLAUDE.md`의 Architecture, Redis 키 구조, Port Assignment, Development Guidelines 섹션
- 검증 대상이 속한 모듈의 `docs/Systems/` 문서 (`docs/Documentation-Checklist.md` 매핑 참조)

문서가 없거나 비어있으면 사용자에게 알리고 중단한다.

### 2단계: 워크트리 격리 에이전트 실행

Agent를 호출하여 검증을 수행한다. 반드시 다음 파라미터를 설정:
- `isolation: "worktree"` -- 워크트리 격리 (필수)
- `subagent_type: "general-purpose"`
- `description: "시스템 설계 검증"`

에이전트 프롬프트 구조:
```
당신은 시스템 설계 정합성 검증 전문가입니다.

## 검증 대상
[1단계에서 수집한 데이터: diff 전문 / 계획 내용 / 설명 텍스트]

## 설계 문서
[1단계에서 수집한 CLAUDE.md 관련 섹션 + docs/Systems/ 문서 전문]

## 지시사항
1. 검증 대상을 설계 문서의 각 영역과 대조한다.
2. 검증 항목:
   - 모듈 경계: common/room/chat 분리, 공유 코드는 common (위반 예: room에서 chat 직접 참조)
   - Redis 키 설계: CLAUDE.md 키 패턴, TTL, 타입 규칙 (위반 예: 미등록 키, TTL 미설정)
   - 포트 할당: CLAUDE.md Port Assignment 테이블 (위반 예: 미등록 포트, 충돌)
   - Protobuf 프로토콜: proto/ 정의, Envelope 패턴 (위반 예: Envelope 우회 직접 직렬화)
   - 내부 채널 설계: Room<->Game, Room->Chat 채널 구조 (위반 예: 채널 역방향 사용)
   - 인증/보안: 토큰 검증 필수, Rate Limiting 필수 (위반 예: 미인증 허용)
   - 스레드 안전성: strand 기반 동시성, RAII 필수 (위반 예: strand 미사용 공유 접근)
   - 로그 형식: Vector 파싱 정규식 일치 형식 (위반 예: 비표준 로그 형식)
   - 인프라 정합성: Docker Compose 서비스/포트/의존성 (위반 예: 미반영 설정 변경)
   - 설정 관리: Config 클래스 접근, .env 관리 (위반 예: 설정값 하드코딩)
3. 코드 품질/성능은 판단하지 않는다. 오직 설계 의도와의 정합성만 본다.
4. 설계 문서에 명시되지 않은 내용은 추측하지 않는다.
5. "차후 추가 예정" 또는 미정의 영역은 위반이 아닌 "미정의"로 분류한다.

## 심각도 기준
- 위반: 설계 원칙에 명시적으로 반하는 구현 (모듈 순환 의존, 미인증 접근, RAII 미적용)
- 주의: 설계 의도와 미묘하게 어긋날 수 있는 구현 (공유 코드가 common 대신 room에 위치)

### 최종 판단
```

### 3단계: 결과 출력 및 후속 안내

에이전트 결과를 그대로 출력한다.

- **설계 부합**: 결과만 출력하고 종료.
- **설계 불일치**: 위반 항목을 나열하고, 설계에 맞는 대안을 제안한다. 코드 수정이 필요하면 직접 수정하지 않고 사용자에게 안내한다.
- **설계 미정의**: 해당 영역의 설계를 docs/Systems/ 문서에 추가할 것을 권장한다.

## 주의사항

- **설계 문서가 최우선**: 코드가 설계와 다르면, 설계가 맞는지 코드가 맞는지 판단하지 않는다. 불일치 사실만 보고하고 사용자가 판단한다.
- **코드 수정 금지**: 이 스킬은 검증만 수행한다. 수정이 필요하면 사용자에게 안내한다.
- **과잉 해석 금지**: docs/Systems/ 및 CLAUDE.md에 명시적으로 기술된 내용만 기준으로 삼는다. 문서에 없는 설계 의도를 추측하지 않는다.
- **미정의 영역 존중**: 문서에 정의되지 않은 영역은 위반으로 판단하지 않고 "미정의"로 분류한다.
- **에이전트 프롬프트에 데이터 직접 포함**: 설계 문서/diff를 프롬프트에 포함하여 에이전트가 추가 파일을 읽는 횟수를 최소화한다.
