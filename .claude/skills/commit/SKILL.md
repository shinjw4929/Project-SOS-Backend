## 주제별 분류 커밋

### 1단계: 현재 변경사항 파악

다음 명령을 병렬로 실행하여 변경 내용을 파악한다:
- `git status` (untracked 파일 확인, `-uall` 플래그 사용 금지)
- `git diff` (unstaged 변경사항)
- `git diff --cached` (staged 변경사항)
- `git log --oneline -5` (최근 커밋 스타일 참고)

### 2단계: 주제별 분류

변경된 파일들을 **논리적 주제(topic)** 기준으로 그룹화한다.

**분류 기준** (우선순위 순):
1. **기능 단위**: 같은 기능/모듈에 속하는 변경은 하나의 그룹 (예: Room 관련 파일들)
2. **변경 성격**: 버그 수정, 새 기능, 리팩토링, 문서, 설정 변경 등 성격이 다르면 분리
3. **의존성**: 서로 의존하는 변경은 같은 그룹에 포함 (예: proto 정의 변경 + 핸들러 수정)

**분류 규칙**:
- docs/ 문서 변경은 해당 코드 변경과 같은 그룹에 포함 (코드와 문서가 동일 주제일 때)
- .claude/ 스킬/설정 변경은 별도 그룹으로 분리
- CLAUDE.md 변경은 별도 그룹 또는 관련 코드 변경과 묶기
- infra/ 설정 변경(docker-compose, Vector, Grafana 등)은 관련 코드와 묶거나 별도 그룹
- 단독 문서 변경(코드 변경 없이 문서만 수정)은 하나의 그룹으로 묶기
- 1개 파일만 변경된 주제라도 독립 커밋으로 분리

### 3단계: 커밋 계획 제시

분류 결과를 사용자에게 **표 형태**로 보여주고 확인을 받는다:

```
커밋 #1: <제목>
  - 파일1
  - 파일2

커밋 #2: <제목>
  - 파일3

커밋 #3: <제목>
  - 파일4
  - 파일5
```

사용자가 분류를 조정하면 반영한다. 확인을 받은 후에만 4단계로 진행한다.

### 4단계: 순차 커밋 실행

확인된 계획에 따라 **커밋 번호 순서대로** 하나씩 실행한다:

각 커밋마다:
1. 해당 그룹의 파일만 `git add <파일>` (절대 `git add -A` 사용 금지)
2. HEREDOC 방식으로 커밋 실행

**커밋 메시지 형식**:
```
<제목: 작업 내용 요약 (한 줄)>

<세부 작업 내용>
- 변경된 파일/모듈 목록
- 수정 의도 및 해결한 문제
```

**작성 원칙**:
1. **제목**: 무엇을 했는지 명확하게 요약 (예: "Room 퇴장 시 Redis 세션 정리 로직 추가")
2. **본문**: 제목과 빈 줄 하나 띄우고 세부 내용 작성
3. **의도 명시**: 단순 변경 사항 나열이 아닌, 왜 이 변경이 필요했는지 드러나도록 작성
4. **간결함**: 불필요한 설명 없이 핵심만 기술
5. **한국어 작성**: 제목과 본문 모두 한국어로 작성
6. **Co-Authored-By 금지**: 커밋 메시지에 `Co-Authored-By` 트레일러를 절대 추가하지 않는다

### 5단계: 결과 보고

모든 커밋 완료 후 `git log --oneline -N` (N = 커밋 수)으로 결과를 보여준다.

**금지 사항**:
- `.env`, credentials 등 민감한 파일은 커밋하지 않는다
- push는 사용자가 명시적으로 요청하지 않는 한 하지 않는다
