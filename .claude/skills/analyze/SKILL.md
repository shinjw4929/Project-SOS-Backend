## 분석 실행 (읽기 전용)

이 스킬은 코드를 수정하지 않는다. 분석 결과만 출력한다.

### 1단계: 대상 확인

$ARGUMENTS에서 분석 대상을 파악한다. 대상 유형:
- **모듈**: common, room, chat, infra
- **클래스/함수**: 특정 클래스 또는 메서드
- **Redis 키**: 키 패턴 (token:*, chat:session:* 등)
- **포트**: 서비스 포트 (8080, 8081, 8082, 8083 등)
- **Protobuf 메시지**: Envelope payload, 내부 프로토콜
- **변경 시나리오**: "X를 변경하면 어디가 영향받는가"

### 2단계: 정적 의존성 수집

대상 유형에 따라 탐색한다:

**코드 의존성**:
- `#include` 체인 추적
- 클래스/함수 사용처 검색 (Grep)
- Protobuf 메시지 참조 검색

**Redis 키 의존성**:
- 키 패턴 사용처 검색 (SessionStore, ChannelManager, RateLimiter)
- TTL 설정 확인
- 키 생성/조회/삭제 위치

**포트 의존성**:
- docker-compose.yml 포트 매핑
- Config accessor 사용처
- 내부 채널 연결 설정

**인프라 의존성**:
- Docker 서비스 의존성 그래프
- Vector 소스/싱크 매핑
- Grafana 대시보드 쿼리

### 3단계: 의존성 그래프 구성

수집한 의존성을 그래프로 정리한다:

```
[대상]
├── 직접 의존 (이 코드가 사용하는 것)
│   ├── 모듈/파일
│   └── Redis 키/포트
└── 역방향 의존 (이 코드를 사용하는 것)
    ├── 모듈/파일
    └── 설정/인프라
```

### 4단계: 영향 범위 평가

변경 시나리오가 주어진 경우:
- 직접 수정이 필요한 파일
- 간접적으로 영향받는 파일 (동작 변경 가능성)
- 확인이 필요한 설정/인프라
- 문서 업데이트 대상

### 5단계: 결과 출력

```
## 분석 결과: [대상]

### 의존성 그래프
[그래프]

### 영향 범위
| 파일 | 영향 유형 | 설명 |
|------|----------|------|

### 수정 필요 파일
- [직접 수정 필요한 파일 목록]

### 확인 필요
- [테스트/설정 확인이 필요한 항목]
```
