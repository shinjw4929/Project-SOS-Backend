## 역할

지정된 모듈, 클래스, Redis 키, 포트, 또는 기능 영역의 의존성과 영향 범위를 분석하여 보고한다.
코드 수정은 하지 않는다. 분석 결과만 출력한다.

$ARGUMENTS가 있으면 해당 내용을 분석 대상으로 반영한다. 없으면 `AskUserQuestion` 도구로 분석 유형을 선택지로 제시한다 (모듈 / 클래스 / Redis 키 / 포트 / 변경 시나리오).

## 분석 절차

### 1단계: 분석 대상 확정

사용자가 지정한 대상을 아래 유형 중 하나로 분류한다:

| 유형 | 예시 | 분석 범위 |
|---|---|---|
| **모듈** | common, room, chat | 모듈 내부 구조 + 외부 의존성 |
| **클래스/함수** | `SessionStore`, `RoomManager` | #include 체인, 호출처, 의존 모듈 |
| **Redis 키** | `token:*`, `chat:session:*` | 키 생성/조회/삭제 위치, TTL, 연관 모듈 |
| **포트** | 8080, 8081 | Docker 매핑, Config 참조, 내부 채널 연결 |
| **Protobuf 메시지** | `CreateRoomRequest`, `Envelope` | proto 정의, 직렬화/역직렬화 위치, dispatch 분기 |
| **변경 시나리오** | "SessionStore에 새 메서드 추가" | 영향받는 모듈/설정/테스트 |

### 2단계: 정적 의존성 수집

병렬로 다음을 수집한다:

**A. 코드 의존성** (대상이 클래스/함수인 경우)
- `#include` 체인 추적으로 직접 의존성 파악
- Grep으로 대상 클래스/함수 사용처 전체 수집
- 모듈별 분포 확인 (common/room/chat)

**B. Redis 키 의존성** (대상이 Redis 키인 경우)
- 키 패턴 사용처 검색 (SessionStore, ChannelManager, RateLimiter 등)
- TTL 설정 확인
- 키 생성/조회/삭제 위치와 타이밍

**C. 포트 의존성** (대상이 포트인 경우)
- docker-compose.yml 포트 매핑
- Config accessor 사용처
- 내부 채널(GameServerChannel, ChatServerChannel) 연결 설정

**D. Protobuf 의존성** (대상이 메시지인 경우)
- proto 파일 정의 확인
- Envelope payload_case 분기 위치
- 직렬화(SerializeToString) / 역직렬화(ParseFromString) 호출 위치

**E. 인프라 의존성**
- Docker 서비스 의존성 그래프 (depends_on)
- Vector 소스/싱크 매핑
- Grafana 대시보드 쿼리 참조

### 3단계: 의존성 그래프 구성

수집된 데이터를 기반으로 의존성 그래프를 구성한다:

```
[대상]
├── 직접 의존 (이 코드가 사용하는 것)
│   ├── 모듈/파일
│   ├── Redis 키
│   └── Protobuf 메시지
└── 역방향 의존 (이 코드를 사용하는 것)
    ├── 모듈/파일
    ├── 설정/인프라
    └── 테스트
```

### 4단계: 영향도 평가

**변경 시나리오가 주어진 경우**, 아래 항목을 평가한다:

| 영향 항목 | 확인 내용 |
|---|---|
| **컴파일 영향** | #include 체인, 헤더 변경으로 재빌드 필요한 파일 |
| **런타임 영향** | 호출 체인, 데이터 흐름 변경으로 동작이 바뀌는 모듈 |
| **네트워크 영향** | Protobuf 메시지 변경, 내부 채널 프로토콜 변경 |
| **인프라 영향** | Docker 설정, Redis 스키마, ClickHouse 스키마 변경 |
| **테스트 영향** | 기존 테스트 수정/추가 필요 여부 |

### 5단계: 분석 결과 출력

```
## 의존성 분석 결과

### 분석 대상
- 대상: [모듈/클래스/키/포트]
- 유형: [모듈 / 클래스 / Redis 키 / 포트 / 변경 시나리오]

### 의존성 그래프
(텍스트 다이어그램)

### 접근 맵
| 파일 | 접근 유형 | 모듈 |
|------|-----------|------|
| ...  | #include / 호출 / Redis 접근 | common/room/chat |

### 영향도 평가 (변경 시나리오가 주어진 경우)
| 영향 항목 | 파일 | 설명 |
|-----------|------|------|
| 컴파일    | ...  | ...  |
| 런타임    | ...  | ...  |
| 인프라    | ...  | ...  |

### 주의 사항
- (변경 시 특별히 주의해야 할 점)

### 권장 작업 순서
1. (변경이 필요한 경우, 안전한 수정 순서 제안)
```

## 주의사항

- **코드 수정 금지**: 이 스킬은 분석만 수행한다. 수정이 필요하면 사용자에게 안내하거나 다른 스킬(`/plan-create`, `/debug`)을 권장한다.
- **사실 기반**: Grep/Read로 확인한 내용만 보고한다. 추측은 "미확인"으로 표시한다.
- **Docs 우선 참조**: 분석 시작 전 `docs/Systems/` 관련 문서를 읽어 현재 아키텍처를 파악한 뒤 코드를 탐색한다.
- **모듈 구분 명시**: 모든 파일에 common/room/chat/infra 모듈을 표기한다.
