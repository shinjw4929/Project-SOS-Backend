## 역할

C++ 백엔드 서버에서 발생하는 에러, 비정상 동작, 크래시를 체계적으로 진단하고 수정안을 제시한다.

$ARGUMENTS가 있으면 해당 내용을 에러 메시지/증상/재현 조건으로 반영한다.

## 디버깅 절차

### 1단계: 증상 수집

사용자로부터 다음 정보를 확보한다. $ARGUMENTS에서 추출할 수 없는 항목은 사용자에게 질문한다.
**발생 위치**가 불명확한 경우 `AskUserQuestion` 도구로 선택지를 제시한다 (Room Server / Chat Server / common / infra).

| 항목 | 설명 |
|---|---|
| **에러 메시지** | 로그 출력, 스택 트레이스, 에러 코드 |
| **증상** | 예상 동작 vs 실제 동작 |
| **재현 조건** | 언제, 어떤 상황에서 발생하는지 |
| **발생 위치** | 어떤 서버/모듈에서 발생하는지 |

### 2단계: 에러 유형 분류 및 1차 탐색

에러를 아래 카테고리로 분류하고, 해당 카테고리의 점검 항목을 우선 확인한다.

**A. 메모리/RAII 에러**
- segfault, double free, use-after-free: shared_ptr/unique_ptr 수명 추적, 소켓/세션 닫힘 순서 확인
- 순환 참조: shared_ptr 간 순환 → weak_ptr 전환 필요 여부
- dangling reference: 비동기 콜백에서 이미 파괴된 객체 접근 → enable_shared_from_this 패턴 확인

**B. 스레드/동시성 에러**
- data race, 간헐적 크래시: io_context 스레드 모델 확인, strand 사용 여부
- 교착 상태: mutex 잠금 순서, async 체인 내 동기 호출 여부
- 경합 조건: 비동기 콜백 간 공유 상태 접근 패턴

**C. 네트워크 I/O 에러**
- 연결 끊김, 타임아웃: async_read/write 에러 핸들링, Boost.Asio error_code 처리
- 메시지 유실: Codec 버퍼 관리, length-prefixed 프레이밍 정확성
- 내부 채널 연결 실패: GameServerChannel/ChatServerChannel 연결 설정 확인

**D. Protobuf 직렬화 에러**
- 파싱 실패: proto 정의와 코드 불일치, Envelope payload_case 분기 누락
- 필드 누락: 필수 필드 미설정, 기본값 의존
- 잘못된 dispatch: processMessage 내 case 분기 오류

**E. Redis 연결 에러**
- 토큰 검증 실패: SessionStore 키 패턴, TTL 만료, 키 생성/조회 타이밍
- Rate Limit 오작동: RateLimiter 윈도우 계산, 키 패턴 충돌
- 키 미정리: 세션 종료 시 Redis 키 삭제 누락, TTL 미설정

**F. Docker/인프라 에러**
- 컨테이너 시작 실패: docker-compose.yml 구문, 이미지 빌드 에러
- 서비스 간 연결 불가: 포트 매핑, 네트워크 설정, depends_on 순서
- 로깅 파이프라인 장애: Vector 소스/싱크, ClickHouse 스키마 불일치

### 3단계: 원인 추적

1. **스택 트레이스 분석**: 에러에 파일명/라인이 있으면 해당 코드를 Read
2. **관련 모듈 탐색**: 에러가 발생한 클래스/함수를 Grep으로 추적하여 호출 체인 파악
3. **정상 흐름 대조**: `docs/Systems/` 관련 문서에서 정상 동작 흐름을 확인하고 실제 코드와 비교
4. **데이터 흐름 추적**: 문제 지점의 입력 → 처리 → 출력을 순서대로 추적

2단계에서 분류한 카테고리별 점검 항목을 체크리스트로 순차 확인하며, 원인을 좁혀간다.

### 4단계: 진단 결과 출력

```
## 디버깅 결과

### 증상 요약
- (에러 메시지 / 비정상 동작 요약)

### 원인 분석
| # | 원인 | 근거 (파일:라인) | 카테고리 |
|---|------|------------------|----------|
| 1 | ...  | ...              | A~F      |

### 수정안
| # | 파일:라인 | 현재 코드 | 수정 코드 | 설명 |
|---|-----------|-----------|-----------|------|
| 1 | ...       | ...       | ...       | ...  |

### 추가 확인 필요
- (수정 후에도 검증이 필요한 사항)
```

### 5단계: 수정 적용

사용자에게 수정 적용 여부를 확인한 뒤:
- **단순 라인 수정**: Edit 도구로 직접 수정한다.
- **새 클래스/모듈 추가 등 큰 변경**: `docs/Checklists/pattern-search-guide.md`를 따라 기존 패턴을 탐색한 뒤 수정한다. 또는 `/implement`로 패턴 기반 구현을 권장한다.
- 수정 적용 후 `/review-comments`를 자동 호출하여 주석 정합성을 점검한다.
- `/build`로 컴파일 검증을 권장한다.
- 수정이 서비스 동작에 영향을 주는 경우 `/test`도 권장한다.

## 주의사항

- **사실 기반 진단**: 코드를 직접 읽고 확인한 내용만 원인으로 제시한다. 추측은 "가능성"으로 명시하고, 확인 방법을 함께 안내한다.
- **최소 변경**: 에러 수정에 필요한 최소한의 변경만 제안한다. 주변 코드 리팩토링은 하지 않는다.
- **CLAUDE.md 준수**: 수정 시 Development Guidelines(RAII, smart pointer, strand 등)를 따른다.
- **근본 원인 우선**: 증상만 가리는 workaround보다 근본 원인 수정을 우선한다. workaround가 필요한 경우 이유를 명시한다.
