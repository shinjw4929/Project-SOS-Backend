# 구현 우선순위

> 2026-03-17 기준. 현재 코드베이스 상태와 문서 간 의존성을 분석하여 도출한 구현 순서.

---

## 설계 요약

- **방 기반 시스템**: 클라이언트가 방을 생성하고, 다른 플레이어가 방 목록에서 선택하여 입장, 전원 준비 후 호스트가 게임 시작
- **1~8인 가변 인원**: PvE 협동 Wave Defense RTS
- **강퇴 기능 없음**: 호스트가 불만족 시 방을 재생성
- **방 목록**: 모든 클라이언트에게 방 목록 공개

## Proto 구조 참조

### room.proto (sos.room)

| 방향 | 메시지 |
|------|--------|
| Client -> Room Server | CreateRoomRequest, JoinRoomRequest, LeaveRoomRequest, ToggleReadyRequest, StartGameRequest, RoomListRequest, RoomHeartbeat |
| Room Server -> Client | CreateRoomResponse, JoinRoomResponse, RoomListResponse, RoomUpdate, GameStart, RejectResponse |
| Internal (Game <-> Room) | TokenValidateRequest/Response, SlotReleased, GameServerHeartbeat |
| 공유 타입 | RoomState, PlayerInfo (player_id, player_name, is_ready, is_host), RoomInfo |
| 프레이밍 | Envelope (oneof) |

### chat.proto (sos.chat)

| 항목 | 내용 |
|------|------|
| 채널 | LOBBY, ALL, WHISPER |
| Internal | SessionCreated, SessionEnded |

## 우선순위

### Phase 1: 공통 기반 scaffolding

> Room/Chat 서버가 모두 미구현이므로 처음부터 공유 구조로 설계한다.

| Step | 작업 | 산출물 | 의존 |
|------|------|--------|------|
| 1-1 | Protobuf 스키마 작성 | `proto/room.proto`, `proto/chat.proto` | - |
| 1-2 | CMake 빌드 체계 구성 | `CMakeLists.txt` (common + room + chat 3타겟, protoc 코드젠) | 1-1 |
| 1-3 | common 라이브러리 구현 | `Codec<T>`, `RedisClient`, `RateLimiter`, `Config`, `Logger` | 1-2 |

**Phase 1 완료.**

**완료 기준**: `cmake --build` 성공, protoc 코드젠 정상, common 라이브러리 빌드 통과.

---

### Phase 2: Room Server 핵심

| Step | 작업 | 산출물 | 의존 |
|------|------|--------|------|
| 2-1 | TCP 서버 골격 | `RoomServer`, `ClientSession` (:8080) | 1-3 |
| 2-2 | Protobuf Envelope 통합 | 4byte LE 프레이밍 + oneof 디스패치 | 2-1 |
| 2-3 | 방 관리 로직 | `RoomManager`, `Room` (생성/입장/퇴장/준비/시작, 방 목록) | 2-2 |

**Phase 2 완료.**

**완료 기준**: TCP 클라이언트가 방 생성 -> 다른 클라이언트 입장 -> 전원 준비 -> 호스트 GameStart 수신.

---

### Phase 3: Room Server Redis 연동

| Step | 작업 | 산출물 | 의존 |
|------|------|--------|------|
| 3-1 | 세션/토큰 관리 | `SessionStore`, `TokenGenerator`, Redis TTL | 2-3 |
| 3-2 | Rate Limiting | IP 기반 요청 제한 (Redis INCR + EXPIRE) | 2-3 |
| 3-3 | 내부 TCP 채널 | `GameServerChannel` (:8081, 토큰 검증 + SlotReleased + 하트비트) | 3-1 |

**Phase 3 완료.**

**완료 기준**: 토큰 발급 -> 검증 -> TTL 만료, Rate Limit 동작, 하트비트 기반 세션 복구.

---

### Phase 4: Chat Server

| Step | 작업 | 산출물 | 의존 |
|------|------|--------|------|
| 4-1 | Chat Server 핵심 | `ChatServer`, `ChatSession` (:8082), ChatAuth, LobbyChannel | 1-3 |
| 4-2 | 세션 채널 통합 | `InternalChannel` (:8083), SessionCreated/Ended, ALL 채널, 히스토리 | 4-1, 3-3 |
| 4-3 | Room Server에 ChatServerChannel 추가 | Room -> Chat 서버 SessionCreated/SessionEnded 전송 | 4-2 |

**완료 기준**: 로비 채팅 + 게임 시작 후 세션 채널 전환 + ALL 브로드캐스트 동작.

> 4-1은 Phase 1 완료 후 Phase 2~3과 병렬 진행 가능. 단, 4-2는 Room Server의 내부 채널(3-3)에 의존.

---

### Phase 5: 로깅 통합 + 운영

| Step | 작업 | 산출물 | 의존 |
|------|------|--------|------|
| 5-1 | C++ 서버 로그 출력 형식 적용 | spdlog 커스텀 포매터 (공통 로그 형식) | 2-1, 4-1 |
| 5-2 | Docker Compose 서버 활성화 | room-server/chat-server 주석 해제, Dockerfile 작성 | 5-1 |
| 5-3 | Room/Chat Grafana 대시보드 추가 | Room Metrics, Chat Metrics, Error Dashboard | 5-2 |
| 5-4 | 설정 파일 + Graceful Shutdown | `server_config.json`, SIGINT 핸들링 | 4-3 |
| 5-5 | Grafana 알림 규칙 | 에러 급증, 방 생성 실패, Redis 장애 | 5-3 |

**완료 기준**: `docker compose up`으로 전체 스택 기동, 로그 자동 수집, 대시보드 표시.

---

### Phase 6: 테스트 + Unity 연동

| Step | 작업 | 산출물 | 의존 |
|------|------|--------|------|
| 6-1 | 유닛 테스트 | Catch2 (RoomManager, Room, Codec, TokenGenerator, RateLimiter, Config, ChannelManager) | 5-4 |
| 6-2 | 피처 테스트 | Catch2 (TCP E2E, Redis 연동, 내부 채널, 방 생명주기 엣지 케이스) | 6-1 |
| 6-3 | 부하 테스트 | Python asyncio 더미 클라이언트, benchmark.md | 6-2 |
| 6-4 | Unity 클라이언트 연동 | RoomClient.cs, ChatClient.cs, 토큰 검증, 방 목록 UI, 채팅 UI | 5-4 |

**완료 기준**: 전체 테스트 통과, Unity 다수 클라이언트 방 생성 -> 입장 -> 준비 -> 게임 시작 -> 채팅 E2E 동작.

## 의존성 그래프

```
[Phase 1] 공통 기반
    │
    ├──► [Phase 2] Room 핵심 ──► [Phase 3] Room Redis
    │                                   │
    │                                   ▼
    └──► [Phase 4-1] Chat 핵심 ──► [Phase 4-2~3] Chat 세션 채널
                                        │
                                        ▼
                                  [Phase 5] 로깅 통합 + 운영
                                        │
                                        ▼
                                  [Phase 6] 테스트 + Unity 연동
```

- Phase 1은 모든 작업의 선행 조건 (1-1 Protobuf 스키마 완료됨)
- Phase 2~3 (Room)과 Phase 4-1 (Chat 핵심)은 Phase 1 완료 후 병렬 가능
- Phase 4-2 (Chat 세션 채널)은 Phase 3-3 (Room Server 내부 채널)에 의존
- Phase 5~6은 서버 구현 완료 후 순차 진행

## 핵심 제약

1. **Protobuf 스키마 완료**: 스키마가 확정되어 Codec, 서버 로직, Unity 연동 작업 착수 가능
2. **Room Server가 Chat보다 선행**: Chat의 세션 채널이 Room Server의 SessionCreated/SessionEnded에 의존
3. **채팅 채널**: LOBBY, ALL, WHISPER
4. **강퇴 없음**: 호스트가 방을 재생성하는 방식으로 대체. 불필요한 권한 로직 배제
5. **로깅 인프라는 준비 완료**: C++ 서버가 정의된 stdout 형식만 맞추면 파이프라인 자동 편입
6. **docker-compose의 서버 서비스는 현재 주석 처리**: Dockerfile + 빌드가 준비되면 활성화
