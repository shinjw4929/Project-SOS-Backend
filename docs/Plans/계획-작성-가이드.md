# 계획 작성 가이드

---

## 1. 계획 요청 시 필요 정보

| 항목 | 설명 |
|------|------|
| **변경 대상** | 어떤 모듈/서비스/파일을 변경하는가 |
| **현재 문제** | 지금 무엇이 부족하거나 잘못되어 있는가 |
| **원하는 결과** | 변경 후 어떤 상태가 되어야 하는가 |
| **제약 조건** | 기한, 호환성, 성능 요구사항, 외부 의존성 |

---

## 2. 계획 문서 구조

계획은 3종류의 파일로 구성된다.

### orchestration.md (중심 문서)

전체 계획의 요약과 Phase 간 관계를 정의한다.

```markdown
# [계획명] — Orchestration

## 문제 정의
[현재 문제와 변경 동기]

## AS-IS
[현재 상태: 관련 코드/설정/인프라의 구조]

## TO-BE
[목표 상태: 변경/추가/삭제 항목]

## Phase 체크리스트

| Phase | 파일 | 목표 | 상태 |
|-------|------|------|------|
| 1 | phase-1-xxx.md | ... | 대기 |
| 2 | phase-2-xxx.md | ... | 대기 |

## Phase 의존성
[Phase 간 선후 관계, 병렬 가능 여부]

## 검증 방법
[전체 완료 후 검증 절차]

## 롤백 계획
[Phase별 롤백 가능 단위]
```

### phase-N-[제목].md (Phase 파일)

각 Phase의 상세 실행 계획.

```markdown
# Phase N: [제목]

## 목표
[이 Phase에서 달성할 것]

## 전제 조건
[선행 Phase 또는 외부 조건]

## 작업 목록

### 병렬 작업 (독립적, 동시 진행 가능)
- [ ] 작업 A
- [ ] 작업 B

### 순차 작업 (의존성 있음)
- [ ] 작업 C (작업 A 완료 후)
- [ ] 작업 D

## 변경 파일

| 파일 | 변경 내용 |
|------|----------|
| ... | ... |

## 검증
- [ ] 빌드 성공
- [ ] 테스트 통과
- [ ] [Phase 특화 검증 항목]
```

### execution-log.md (실행 기록)

계획 실행 과정의 진행 상황을 추적한다.

```markdown
# [계획명] — Execution Log

| Phase | 시작 | 완료 | 결과 | 비고 |
|-------|------|------|------|------|
| 1 | - | - | 대기 | |
| 2 | - | - | 대기 | |
```

## 3. 계획 작성 주의사항

1. **문제를 먼저 정의**: AS-IS를 코드/설정에서 직접 확인. 추측하지 않는다
2. **Phase는 롤백 안전 단위**: 각 Phase 완료 후 시스템이 정상 동작해야 함
3. **병렬 작업 분리**: 의존성 없는 작업은 병렬로 표기
4. **Phase별 검증**: 매 Phase 완료 시 빌드 + 관련 테스트 실행
5. **역호환성**: 기존 기능을 깨지 않는 순서로 Phase 배치

## 4. 디렉토리 구조

```
docs/Plans/
├── 계획-작성-가이드.md        # 본 문서
├── 구현 우선순위.md           # 전체 구현 로드맵
├── [주제]/                   # 활성 계획
│   ├── orchestration.md
│   ├── phase-1-xxx.md
│   ├── phase-2-xxx.md
│   └── execution-log.md
└── Completed/                # 완료된 계획 아카이브
    └── [주제]/
```

---

## 5. 완료 후 처리

1. execution-log.md에 최종 결과 기록
2. `docs/Systems/` 관련 문서 업데이트 (`/update-docs`)
3. 계획 폴더를 `docs/Plans/Completed/`로 이동 (필수)
