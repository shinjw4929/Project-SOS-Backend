# 채팅 서버 구축 계획

> **목적**: 로비/인게임 전 구간에서 채팅이 가능한 독립 채팅 서버 구축.
> **게임 특성**: PvE 협동 Wave Defense RTS (1~8인)
> **배포 환경**: 자체 서버/VPS + Docker
> **프로토콜 정의**: [`chat.proto`](chat.proto) (Single Source of Truth)

---

## 기술 스택 선택: C++ (룸 서버와 동일)

| 기준 | C++ 동일 스택 | Go/Node.js 별도 스택 |
|------|--------------|---------------------|
| 코드 재사용 | Codec, Redis 래퍼, RateLimiter 전부 재사용 | 전부 새로 작성 |
| 프로토콜 | Envelope oneof 패턴 확장 | 별도 프레이밍 구현 |
| 빌드/배포 | CMake + vcpkg + 동일 Docker base | 별도 빌드 체인 |
| 운영 | Redis 1개, C++ 프로세스 2개 | Redis + 별도 런타임 |

**결론**: 룸 서버와 동일 저장소(Project-SOS-Backend) 내에 별도 바이너리로 구축. 공통 코드는 CMake STATIC 라이브러리로 공유.

> **전제**: 룸 서버 코드가 아직 미구현 상태이므로, 처음부터 common 라이브러리 구조로 설계하여 룸/채팅 서버를 동시에 scaffolding한다. "리팩토링"이 아닌 "초기 설계 시 공유 구조 반영"이다.

## 프로토콜: TCP + Protobuf

WebSocket은 브라우저 호환이 필요할 때만 의미 있다. Unity 네이티브 클라이언트에서는 TCP + Protobuf가 RoomClient.cs 패턴을 그대로 복제할 수 있어 구현 비용이 최소화된다.

### 포트 할당

```
8080  룸 서버 (클라이언트 대면)
8081  룸 서버 (내부: 게임 서버 연결)
8082  채팅 서버 (클라이언트 대면)
8083  채팅 서버 (내부: 룸 서버/게임 서버 연결)
7979  게임 서버 (Netcode for Entities)
6379  Redis
```

## chat.proto 스키마

**파일**: Proto/chat.proto (Project-SOS-Protocol submodule에 추가)
**래퍼**: 독립적인 ChatEnvelope 사용 (room.proto의 Envelope와 분리 -- 서버 간 프로토콜 결합 방지)
**프레이밍**: 동일 (4byte 길이 LE + Protobuf binary)

```protobuf
syntax = "proto3";
package sos.chat;

enum ChatChannel {
    CHANNEL_UNKNOWN = 0;
    CHANNEL_LOBBY = 1;       // 로비 전체
    CHANNEL_ALL = 2;         // 인게임 전체
    CHANNEL_WHISPER = 3;     // 1:1 귓속말
}

// 클라이언트 -> 채팅 서버
message ChatAuth {
    string player_id = 1;
    string session_id = 2;
    string player_name = 3;  // 표시명 (ChatReceive.sender_name에 사용)
}
message ChatSend { ChatChannel channel = 1; string content = 2; string whisper_target = 3; }
message JoinChannel { ChatChannel channel = 1; }
message LeaveChannel { ChatChannel channel = 1; }
message ChatHeartbeat {}

// 채팅 서버 -> 클라이언트
message ChatAuthResult { bool success = 1; string reason = 2; }
message ChatReceive {
    ChatChannel channel = 1; string sender_id = 2; string sender_name = 3;
    string content = 4; uint64 timestamp = 5;
}
message SystemMessage { ChatChannel channel = 1; string content = 2; uint64 timestamp = 3; }
message ChatError {
    ChatErrorCode code = 1; string message = 2;
    enum ChatErrorCode {
        UNKNOWN = 0;
        NOT_AUTHENTICATED = 1;
        RATE_LIMITED = 2;
        INVALID_CHANNEL = 3;
        MESSAGE_TOO_LONG = 4;
        PLAYER_NOT_FOUND = 5;
        CHANNEL_NOT_JOINED = 6;  // 참가하지 않은 채널로 전송 시도
    }
}

// 내부 서버 간 (룸 서버 -> 채팅 서버)
message SessionCreated {
    string session_id = 1;
    repeated PlayerInfo players = 2;
    message PlayerInfo {
        string player_id = 1;
        string player_name = 2;  // 표시명 전달
    }
}
message SessionEnded { string session_id = 1; }

message ChatEnvelope {
    oneof payload {
        ChatAuth auth = 1; ChatSend send = 2; JoinChannel join_channel = 3;
        LeaveChannel leave_channel = 4; ChatHeartbeat heartbeat = 5;
        ChatAuthResult auth_result = 10; ChatReceive receive = 11;
        SystemMessage system_message = 12; ChatError error = 13;
        SessionCreated session_created = 20; SessionEnded session_ended = 21;
    }
}
```

### sender_name 전달 경로

```
로비 채팅:
  클라이언트 -> ChatAuth(player_id, "", player_name) -> 채팅 서버
  채팅 서버: ChatAuth.player_name 저장 -> ChatReceive.sender_name으로 사용

인게임 채팅:
  룸 서버 -> SessionCreated(session_id, [{player_id, player_name}, ...]) -> 채팅 서버
  채팅 서버: PlayerInfo.player_name 저장 -> ChatReceive.sender_name으로 사용
  클라이언트 -> ChatAuth(player_id, session_id, player_name) -> 재인증
```

## 채널/룸 관리

```
게임 단계        채널 타입          범위
--------------------------------------------
로비             CHANNEL_LOBBY     전체 접속자
인게임           CHANNEL_ALL       같은 session_id 전체
어디서든         CHANNEL_WHISPER   1:1
```

### 채널 생명주기

1. **로비**: 서버 시작 시 생성. ChatAuth 성공 시 자동 참가, 게임 시작 시 자동 퇴장
2. **세션 채널**: `SessionCreated` 내부 메시지 수신 시 ALL 채널 생성. `SessionEnded` 시 해제
3. **귓속말**: 채널 없음. player_id로 직접 전달

### 메시지 히스토리 정책

| 채널 | 참가 시 이전 메시지 | 구현 |
|------|---------------------|------|
| LOBBY | 표시하지 않음 (참가 시점부터) | 서버 메모리 버퍼 불필요 |
| ALL | 최근 20개 전달 | Redis LIST `chat:history:{session_id}:ALL` TTL 7200s |
| WHISPER | 표시하지 않음 | 서버 저장 불필요 |

세션 채널은 게임 중 재접속 시 맥락 파악을 위해 히스토리가 필요하다. 로비/귓속말은 일회성이므로 불필요.

---

## 메시지 검증 규칙

| 항목 | 규칙 | 에러 코드 |
|------|------|-----------|
| 최대 길이 | 200자 (UTF-8 기준) | MESSAGE_TOO_LONG |
| 빈 메시지 | content가 빈 문자열 또는 공백만 -> 거부 | MESSAGE_TOO_LONG |
| Rate Limit | 5초당 10메시지 | RATE_LIMITED |
| 미참가 채널 전송 | 참가하지 않은 채널로 ChatSend -> 거부 | CHANNEL_NOT_JOINED |
| UI Sanitization | 클라이언트 측 TextMeshPro는 Rich Text를 기본 파싱. `<` 문자 이스케이프 또는 Rich Text 비활성화 | (클라이언트 처리) |

## 서버 간 통신

```
게임 시작 시:
  룸 서버 -> GameStart -> 클라이언트들
  룸 서버 -> SessionCreated(session_id, [{A,nameA},{B,nameB},...]) -> 채팅 서버 :8083

게임 종료 시:
  게임 서버 -> SlotReleased -> 룸 서버
  룸 서버 -> SessionEnded(session_id) -> 채팅 서버 :8083
```

### Redis 키 구조

```
chat:session:{session_id}     -> HASH {player_id: "1", ...}    TTL 7200s
chat:player:{player_id}       -> STRING session_id               TTL 7200s
chat:name:{player_id}         -> STRING player_name              TTL 7200s
chat:rate:{player_id}         -> STRING count                    TTL 5s (5초당 10메시지)
chat:history:{session_id}:ALL -> LIST [최근 20개 직렬화 메시지]  TTL 7200s
```

채팅 서버 재시작 시 세션 복구를 위해 Redis에 기록.

### 동일 player_id 중복 접속 처리

동일 player_id로 새 연결이 들어오면 **기존 연결을 kick** (ChatError(NOT_AUTHENTICATED, "Replaced by new connection") 전송 후 close). 새 연결이 인증을 이어받는다. 이유: 클라이언트 크래시 후 재접속 시 기존 연결이 하트비트 타임아웃(90초)까지 남아있을 수 있음.

> 룸 서버가 아직 미구현이므로, 아래 구조를 **최초 scaffolding 시점**부터 적용한다.

│   │   ├── redis/               # RedisClient, SessionStore
│   │   ├── ratelimit/           # RateLimiter
│   │   └── util/                # Config, Logger
│   ├── room/                    # 룸 서버
│   │   ├── main.cpp
│   │   ├── server/              # RoomServer, ClientSession
│   │   ├── room/                # RoomManager, Room (생성/입장/퇴장/준비/시작)
│   │   └── internal/            # GameServerChannel + ChatServerChannel
│   └── chat/                    # 채팅 서버
│       ├── main.cpp
│       ├── server/              # ChatServer, ChatSession

## Unity 클라이언트 변경

### 신규 파일

| 파일 | 위치 | 역할 |
|------|------|------|
| `ChatClient.cs` | `Client/Network/` | 채팅 서버 :8082 TCP 클라이언트 (DontDestroyOnLoad) |
| `ChatUIController.cs` | `Client/Controller/UI/` | 채팅 UI MonoBehaviour |
| `Chat.cs` (generated) | `Client/Network/Generated/` | protoc C# 코드젠 결과 |

### ChatClient.cs 생명주기

```
[앱 시작] -> ChatClient 생성 (DontDestroyOnLoad)
    |-- [로비] 채팅 서버 접속 -> ChatAuth(player_id, "", player_name) -> 로비 채널
    |-- [게임 시작] RoomClient.OnGameStart -> ChatAuth(player_id, session_id, player_name) 재전송
    |   -> 로비 퇴장 + 세션 채널(ALL) 참가
    |-- [인게임] ALL 채팅
    |-- [게임 종료] -> 로비 채널 복귀
    +-- [앱 종료] TCP 연결 종료

RoomClient.cs와 동일한 패턴: async/await + TcpClient, 4byte 프레이밍, ChatEnvelope 파싱.
ECS 연동 불필요: 채팅은 게임 로직에 영향 없음. MonoBehaviour -> MonoBehaviour 이벤트로 충분.
```

### 클라이언트 재접속 전략

채팅은 게임 진행에 필수가 아니므로, 룸 서버보다 관대한 재접속 정책을 적용한다.

```
채팅 서버 연결 끊김 감지
  -> UI 상태 표시 ("채팅 연결 끊김")
  -> 백그라운드 재시도 (exponential backoff: 1s -> 2s -> 4s -> 8s -> 최대 30s)
  -> 재접속 성공 시 ChatAuth 재전송 -> 채널 자동 복귀
  -> 게임 플레이에는 영향 없음 (입력/이동/전투 정상 동작)

룸 서버 재접속과의 차이:
  - 룸 서버: 연결 끊김 = 방 상태 불일치 -> 즉시 에러 UI + 수동 재시도
  - 채팅 서버: 연결 끊김 = 채팅 불가 -> 백그라운드 자동 재시도, 게임 계속
```

### 입력 충돌 처리

```csharp
// ChatUIController.cs
public static bool IsChatFocused;
```

> **UserState.Dead 패턴과의 차이**: UserState는 ECS 싱글톤(`SystemAPI.GetSingleton<UserState>()`)이지만, IsChatFocused는 MonoBehaviour 소스의 managed static bool이다. 채팅 포커스 상태를 ECS 싱글톤으로 만드는 것은 과잉이며, 입력 시스템들이 이미 Burst 미적용(UnityEngine.Input/InputSystem 사용)이므로 managed static 접근이 가능하다.

**키보드 입력만 차단, 마우스 입력은 허용**: RTS에서 채팅 중에도 카메라 이동(마우스 엣지패닝/스크롤줌)과 유닛 선택(클릭/드래그)이 가능해야 UX가 자연스럽다. 따라서 `IsChatFocused` 체크는 키보드 입력 처리 직전에 배치하되, 마우스 입력 로직은 차단하지 않는다.

### 수정 대상 파일 (키보드 입력 사용 시스템 전체)

| 파일 | 키보드 입력 | 변경 내용 |
|------|------------|----------|
| `UnitCommandInputSystem.cs` | A (AttackMove) | IsChatFocused 시 키보드 분기 skip |
| `UserSelectionInputUpdateSystem.cs` | (마우스만) | **변경 불필요** -- 마우스 기반이므로 차단하지 않음 |
| `StructurePlacementInputSystem.cs` | (마우스 + ESC) | ESC 처리 전 IsChatFocused 체크 |
| `StructureCommandInputSystem.cs` | Q, W, E, R | IsChatFocused 시 키보드 분기 skip |
| `ConstructionMenuInputSystem.cs` | Q, W, E, R | IsChatFocused 시 키보드 분기 skip |
| `CameraSystem.cs` | T (모드 토글) | IsChatFocused 시 T키 처리 skip |
| `EntitySelectionSystem.cs` | ESC (선택 해제) | IsChatFocused 시 ESC 처리 skip |

**적용 패턴 예시**:

```csharp
// ConstructionMenuInputSystem.cs OnUpdate
var keyboard = Keyboard.current;
if (keyboard == null) return;
if (ChatUIController.IsChatFocused) return;  // 키보드 전체 차단

// CameraSystem.cs OnUpdate (마우스 이동은 유지, T키만 차단)
if (Keyboard.current != null && Keyboard.current.tKey.wasPressedThisFrame)
{
    if (!ChatUIController.IsChatFocused)  // T키만 조건부 차단
    {
        // 카메라 모드 토글
    }
}
// 마우스 엣지패닝, 스크롤줌 로직은 IsChatFocused 체크 없이 그대로 실행
```

## 구축 단계

### Phase 1: 기반 구조 (초기 scaffolding)

> 룸 서버 코드가 아직 없으므로, Project-SOS-Backend 저장소를 **처음부터 공유 라이브러리 구조**로 생성한다.

```
작업:
1. Project-SOS-Backend 저장소 생성 (CMakeLists.txt: common + room + chat 3개 타겟)
2. Project-SOS-Protocol 저장소에 chat.proto 추가 (room.proto와 함께)
3. Protocol 저장소를 submodule로 추가
4. src/common/ 공유 라이브러리 scaffolding:
   - protocol/Codec.h: 템플릿화 Codec<T> (Codec<Envelope>, Codec<ChatEnvelope>)
   - redis/RedisClient.h: redis-plus-plus 래퍼
   - ratelimit/RateLimiter.h: Redis 기반 Rate Limit
   - util/Config.h, Logger.h
5. protoc 코드젠 통합: room.proto + chat.proto -> C++/C# 코드젠
6. 빌드 확인: cmake --preset=default && cmake --build build
```

**완료 기준**: common 라이브러리 빌드 성공, room-server / chat-server 빈 바이너리 생성.

### Phase 2: 채팅 서버 핵심

```
작업:
1. ChatServer + ChatSession (TCP accept on :8082, 4byte 프레이밍)
2. ChatAuth 인증:
   - 로비: player_id + player_name 저장, 세션 없이 허용
   - 인게임: Redis chat:session:{session_id} 검증
   - 동일 player_id 재접속 시 기존 연결 kick
3. ChannelManager + LobbyChannel (로비 채팅)
4. ChatSend -> 메시지 검증 (길이 200자, 빈 메시지 거부) -> ChatReceive 브로드캐스트
5. Rate Limiting 적용 (5초당 10메시지, chat:rate:{player_id})
6. 하트비트 타임아웃 (90초)
```

**완료 기준**: 2개 TCP 클라이언트 접속 -> 로비 채팅 메시지 송수신.

### Phase 3: 세션 채널 통합

```
작업:
1. InternalChannel (:8083, 룸 서버 TCP 연결 수신)
2. SessionCreated 처리:
   - 세션 채널 생성 (ALL)
   - Redis chat:session:{session_id} 저장
   - Redis chat:history:{session_id}:ALL 초기화
3. SessionEnded 처리:
   - 세션 채널 삭제
   - Redis 키 정리
4. ALL 채널 브로드캐스트 (같은 session_id 전체 플레이어)
5. 로비 -> 세션 채널 전환 로직
6. 메시지 히스토리: 세션 채널 참가 시 최근 20개 전달
7. 룸 서버에 ChatServerChannel 추가 (SessionCreated/SessionEnded 전송)
```

**완료 기준**: 내부 채널로 SessionCreated 수신 -> 세션 채널 생성 -> ALL 채팅 동작.

### Phase 4: Unity 클라이언트

```
작업:
1. chat.proto C# 코드젠 (Client/Network/Generated/)
2. ChatClient.cs 구현:
   - RoomClient.cs 패턴 (async/await + TcpClient, DontDestroyOnLoad)
   - 재접속: exponential backoff (1s -> 2s -> 4s -> ... -> 최대 30s)
   - RoomClient.OnGameStart 연동 -> 채널 전환
3. ChatUIController.cs 구현:
   - Enter 토글 (채팅 입력 활성화/비활성화)
   - 인게임에서는 ALL 단일 채널 (탭 전환 불필요)
   - 50개 메시지 링 버퍼
   - IsChatFocused static bool 관리
   - TextMeshPro Rich Text 비활성화 (injection 방지)
4. 입력 충돌 처리:
   - 키보드 입력 차단: ConstructionMenuInputSystem, StructureCommandInputSystem,
     UnitCommandInputSystem, CameraSystem, EntitySelectionSystem, StructurePlacementInputSystem
   - 마우스 입력은 차단하지 않음 (카메라 이동, 유닛 선택 유지)
```

**완료 기준**: 로비 채팅 + 인게임 ALL 채팅 + 채팅 중 마우스 조작 정상 동작.

### Phase 5: 테스트 + Docker 배포

```
작업:
1. 유닛 테스트 (Catch2):
   - ChannelManager: 채널 참가/퇴장/브로드캐스트
   - Rate Limiting: 카운터 로직
   - 메시지 검증: 길이/빈 메시지/미참가 채널
   - 중복 접속: 기존 연결 kick 확인
2. 피처 테스트:
   - 인증 -> 로비 채팅 -> 게임 시작(SessionCreated) -> 세션 채팅 -> 게임 종료(SessionEnded) -> 로비 복귀 E2E
   - 귓속말: player_id 직접 전달
   - 세션 채널 참가 시 히스토리 20개 수신
   - 동일 player_id 재접속 -> 기존 연결 kick
   - 채팅 서버 재시작 -> Redis에서 세션 복구
3. Docker 멀티스테이지 빌드 (빌드 스테이지 공유)
4. docker-compose (Redis + 룸 서버 + 채팅 서버)
```

**완료 기준**: 전체 테스트 통과 + docker-compose up으로 3개 서비스 일괄 기동.
