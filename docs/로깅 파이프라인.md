# 로깅 파이프라인

> 이 문서는 Vector → ClickHouse → Grafana 로깅 파이프라인의 상세 설계를 정의한다.
> 구축 계획 및 배경은 [로깅 시스템 구축 계획](Plans/로깅/로깅%20시스템%20구축%20계획.md)을 참조.

## 전체 흐름

```
[Game Server (Unity)]
  │ persistentDataPath/Logs/{yyyy-MM-dd}/game_*.log
  │ 로컬 파일 (10MB x 5 롤링)
  ▼
[Vector: game_logs source] ── file tail ──┐
                                          │
[Room Server (C++)]                      │
  │ Docker stdout                         ▼
  ▼                              [Vector: parse_game]
[Vector: docker_logs source] ─┐           │ 정규식 파싱
                              │           │ 필드 추출
[Chat Server (C++)]           │           ▼
  │ Docker stdout             │  [Vector: clean_fields] ◄── [Vector: parse_cpp]
  ▼                           │           │                        ▲
  └───────────────────────────┘           │                        │
      label: vector.service               │                   서비스별 필드 추출
                                          ▼
                                 [Vector: clickhouse sink]
                                    batch 1000 / 5s
                                    disk buffer 512MB
                                          │
                                          ▼
                                 [ClickHouse: service_events]
                                    MergeTree, 90일 TTL
                                    MV: violation_rate, error_rate
                                          │
                                          ▼
                                 [Grafana: 대시보드]
                                    Service Overview (9패널)
                                    Game Events (7패널)
                                    Room Metrics (8패널)
                                    Chat Metrics (8패널)
                                    refresh 30s
```

---

## Sources

### game_logs (파일)

```toml
[sources.game_logs]
type = "file"
include = ["/logs/game/**/*.log"]
read_from = "beginning"
```

- Unity Game Server의 로컬 로그 파일을 tail
- Docker 볼륨: `${GAME_LOG_PATH}:/logs/game:ro`
- 체크포인트: `vector_data:/var/lib/vector` 볼륨으로 영속화 (재시작 시 중복 수집 방지)

### docker_logs (Docker)

```toml
[sources.docker_logs]
type = "docker_logs"
include_labels = ["vector.service"]
```

- Docker 컨테이너의 stdout/stderr 수집
- `vector.service` 라벨이 있는 컨테이너만 대상
- Docker 소켓: `/var/run/docker.sock:/var/run/docker.sock:ro`

## Transforms

### parse_game (Game Server 파싱)

```
입력: game_logs
정규식: ^<ts> | <level> | [<world>:<category>] <message>
```

| 추출 필드 | 원본 | 변환 |
|-----------|------|------|
| timestamp | `2026-03-16 12:34:43.415` | `parse_timestamp` (DateTime64) |
| service | - | 고정값 `"game"` |
| level | `INFO   ` | `upcase(strip_whitespace)` |
| world | `S` 또는 `C` | 그대로 |
| category | `Network` | 그대로 |
| message | `Client connected, networkId=1` | `strip_whitespace` |
| network_id | `networkId=1` | `parse_regex` → `to_int` (없으면 0) |
| ghost_id | `idx=3` | `parse_regex` → `to_int` (없으면 0) |

파싱 실패 시 `abort` → 이벤트 드롭 (잘못된 형식의 로그 제거).

환경변수로 주입되는 메타데이터:

| 필드 | 환경변수 | 기본값 |
|------|---------|--------|
| build_version | `BUILD_VERSION` | dev |
| session_id | `SESSION_ID` | local |
| server_id | `SERVER_ID` | dev-1 |

### parse_cpp (C++ Server 파싱)

```
입력: docker_logs
서비스: .label."vector.service" (room / chat)
정규식: ^<ts> | <level> | [<category>] <message>  (category: [\w:]+, 콜론 포함 카테고리 지원)
```

Game Server와 동일한 타임스탬프/레벨 파싱 후, 서비스별 필드 추출:

C++ 서버 전용 필드는 각 서버 구현 시 추가 예정.

### clean_fields (메타데이터 제거)

```
입력: parse_game, parse_cpp
```

ClickHouse에 불필요한 Docker/Vector 메타데이터를 제거:

```
삭제: .file, .host, .source_type, .label, .stream,
      .container_created_at, .container_id, .container_name, .image
```

## Sink: ClickHouse

```toml
[sinks.clickhouse]
type = "clickhouse"
inputs = ["clean_fields"]
endpoint = "http://clickhouse:8123"
database = "project_sos"
table = "service_events"
auth.strategy = "basic"
auth.user = "default"
auth.password = "${CH_PASSWORD}"

batch.max_events = 1000
batch.timeout_secs = 5

[sinks.clickhouse.buffer]
type = "disk"
max_size = 536870912   # 512 MB
when_full = "block"
```

| 설정 | 값 | 근거 |
|------|------|------|
| batch.max_events | 1000 | ClickHouse는 큰 배치에 최적화 |
| batch.timeout_secs | 5 | 이벤트가 적을 때도 5초 내 전송 보장 |
| buffer.type | disk | Vector 비정상 종료 시 이벤트 유실 방지 |
| buffer.max_size | 512 MB | 버스트 구간 + 일시 장애 대응 |
| buffer.when_full | block | 오버플로우 시 유실 대신 배압(backpressure) |

## ClickHouse 스키마

### service_events (통합 이벤트 테이블)

```sql
CREATE TABLE project_sos.service_events (
    -- 공통 필드
    timestamp       DateTime64(3, 'UTC'),
    service         LowCardinality(String),       -- game / room / chat
    build_version   LowCardinality(String),
    session_id      String DEFAULT '',
    server_id       LowCardinality(String) DEFAULT 'dev',
    level           LowCardinality(String),       -- DEBUG / INFO / WARNING / ERROR
    category        LowCardinality(String),       -- Combat / Wave / Economy / Match / Chat ...
    record_type     LowCardinality(String) DEFAULT 'log',
    message         String,

    -- Game Server 전용
    world           LowCardinality(String) DEFAULT '',  -- S / C
    violation_type  LowCardinality(String) DEFAULT '',
    entity_type     LowCardinality(String) DEFAULT '',
    ghost_id        Int32 DEFAULT 0,
    team_id         Int8 DEFAULT 0,
    network_id      Int32 DEFAULT 0,

    -- 유연한 추가 데이터
    context         String DEFAULT '{}'
) ENGINE = MergeTree()
PARTITION BY (service, toYYYYMMDD(timestamp))
ORDER BY (service, category, timestamp)
TTL toDate(timestamp) + INTERVAL 90 DAY DELETE;
```

| 설계 결정 | 근거 |
|-----------|------|
| 단일 테이블 | 3개 서비스 로그를 한 곳에서 크로스 서비스 조회 |
| `LowCardinality` | 카디널리티 낮은 컬럼 (service, level, category)의 압축/쿼리 성능 |
| `PARTITION BY (service, toYYYYMMDD)` | 서비스별 + 일자별 파티션으로 쿼리/삭제 효율화 |
| `ORDER BY (service, category, timestamp)` | 주요 필터 순서에 맞춘 정렬 키 |
| `TTL 90 DAY` | 자동 만료로 디스크 관리 불필요 |

### Materialized Views

| MV | 원본 조건 | 집계 키 | 용도 |
|----|----------|---------|------|
| violation_rate_by_build_mv | `record_type = 'violation'` | (build_version, violation_type, hour) | 빌드별 위반 추이 |
| error_rate_by_service_mv | `level = 'ERROR'` | (service, category, hour) | 서비스별 에러 추이 |

## Vector 단위 테스트

테스트는 `infra/vector/vector-tests.toml`에 별도 관리하며, Vector의 multi-config 병합으로 실행한다.

### 실행

```bash
vector test --config-toml vector.toml --config-toml vector-tests.toml
```

Docker 환경:
```bash
docker compose cp infra/vector/vector-tests.toml vector:/tmp/vector-tests.toml
docker exec <container> vector test \
  --config-toml /etc/vector/vector.toml \
  --config-toml /tmp/vector-tests.toml
```

### 테스트 목록 (7개)

| 테스트명 | 주입 지점 | 검증 내용 |
|---------|----------|----------|
| game_info_with_network_id | parse_game → clean_fields | INFO 파싱, networkId 추출 |
| game_debug_with_ghost_id | parse_game → clean_fields | DEBUG 파싱, idx(ghost_id) 추출 |
| game_client_side_log | parse_game → clean_fields | Client-side(C) world 파싱, ERROR 레벨 |
| game_no_kvp_fields | parse_game → clean_fields | key=value 없는 로그에서 기본값 0 할당 |
| game_invalid_format_dropped | parse_game | 잘못된 형식 → abort로 드롭 |
| cpp_room_fields_pass_through | clean_fields | Room 필드 통과 + Docker 메타데이터 제거 |
| cpp_chat_fields_pass_through | clean_fields | Chat 필드 통과 + label 제거 |

C++ Server 테스트가 `clean_fields`에 직접 주입하는 이유: Vector test framework의 `log_fields`는 flat 값만 지원하여, Docker label 중첩 객체(`.label."vector.service"`)를 주입할 수 없기 때문이다.

## 파이프라인 보호 설정

| 항목 | 설정 | 근거 |
|------|------|------|
| Vector 체크포인트 영속화 | `vector_data:/var/lib/vector` | 재시작 시 파일 재수집 방지 |
| ClickHouse healthcheck | `clickhouse-client --password <pw> -q 'SELECT 1'` (interval 5s, retries 10, start_period 10s) | 초기화 완료 전 연결 방지 |
| Redis healthcheck | `redis-cli -a <pw> ping` (interval 5s, retries 5) | 기동 완료 확인 |
| 서비스 의존성 | `depends_on: clickhouse: condition: service_healthy` | ClickHouse ready 후 Vector/Grafana 기동 |
| Grafana 데이터소스 비밀번호 | `$CH_PASSWORD` (환경변수) | `.env` 변경 시 자동 반영 |
| Redis 인증 | `--requirepass ${REDIS_PASSWORD}` | 미인증 접근 차단 |

## 성능 고려사항

### 수집 지연

- Vector file source: 기본 polling 주기 1초. 로그 파일 쓰기 후 1~2초 내 ClickHouse 도착
- Docker logs source: 실시간 스트림. 쓰기 즉시 수집
- 전투/웨이브 버스트: 배치(1000이벤트/5초)로 소량 insert 방지

### ClickHouse 부하

- 배치 insert: 1000이벤트 / 5초 단위
- 대시보드 쿼리: 30초 주기 × 16패널 = 0.53 쿼리/초
- MergeTree 파티션: 서비스별 + 일자별로 쿼리 범위 제한

### 디스크 사용

- ClickHouse: 90일 TTL 자동 삭제
- Vector 디스크 버퍼: 최대 512MB (정상 시 거의 비어있음)
- Vector 체크포인트: 수 KB (파일별 오프셋)
